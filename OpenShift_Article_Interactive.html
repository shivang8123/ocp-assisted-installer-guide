This is not a polished marketing post. It's a complete field report — every phase, every issue, every possible outcome, and every decision point documented so you don't have to learn the hard way.

Pre-Installation — What You Cannot Skip

DNS — The #1 Cause of Failed Installations

Before opening the Assisted Installer, validate every DNS record from the bastion. DNS failures are silent killers — the installer may pass pre-flight checks but fail partway through bootstrapping, and tracing it back to DNS can take hours.

DNS Verification from Bastion
# Verify all required DNS records before starting
dig api.shivang.lab.kuberox.net
dig api-int.shivang.lab.kuberox.net
dig +short ocp-master1.shivang.lab.kuberox.net
dig +short ocp-worker1.shivang.lab.kuberox.net
dig -x <master1_IP>   # reverse lookup

# Wildcard apps record (critical for routes)
dig +short test.apps.shivang.lab.kuberox.net

API VIP & Ingress VIP — Getting This Right

These are virtual IPs used by OCP's internal HAProxy load balancer. They must be:

  • ✓Free IPs — not assigned to any existing host, ever
  • ✓In the exact same subnet as your node IPs
  • ✓Different from each other
  • ✓Outside any DHCP pool range
  • ✓Verified by ping before starting (no response = safe)
⚠
If a VIP is already in use: You'll get silent ARP conflicts that cause intermittent API timeouts even after installation completes. These are extremely difficult to diagnose after the fact. Verify before you start.

Static IP + MAC Address (nmstate YAML)

The Assisted Installer uses nmstate YAML format for static IP configuration. For each node you need: IP address, subnet mask, gateway, DNS server, and MAC address. Getting the MAC wrong on VMware means the node gets DHCP instead of static, breaking DNS, breaking everything.

⚡
Pro tip: Build a spreadsheet with all node names, IPs, and MACs before touching the installer. You need all of this during the cluster config step, and hunting mid-setup introduces copy-paste errors that are painful to debug.

Full Pre-Installation Checklist

  • ✓Clean all nodes — remove existing virtual disks or old RAID configurations
  • ✓Reserve and document static IPs for all nodes (masters, workers, bastion)
  • ✓Document MAC addresses for all nodes — needed for NIC assignment in the installer
  • ✓Bastion is up, reachable over SSH, using public key authentication
  • ✓DNS resolves all node hostnames, API VIP, and Ingress VIP (forward + reverse)
  • ✓API VIP and Ingress VIP are free IPs verified by ping
  • ✓Red Hat account credentials and pull secret ready
  • ✓SSH public key from bastion generated and ready to paste

Every Possible Discovery Scenario

SCENARIO A Node doesn't appear after 5+ minutes ▼
Possible causes 1. ISO not attached correctly — Check CD/DVD drive in vSphere settings
2. Boot order not set to CD-ROM first — Fix in VM BIOS/EFI
3. No network connectivity — Check VLAN/port group on VM's NIC
4. DNS not resolving assisted-api.openshift.com — Check upstream DNS
5. Firewall blocking outbound HTTPS port 443 — Agent needs console.redhat.com
Fix Work through causes 1→5 in order. Most common on VMware: boot order wrong or NIC on wrong port group.
SCENARIO B Node shows "Insufficient" status — NIC connectivity error ▼
Symptom Node appears in Host Discovery table but shows Status: Insufficient with "NIC connectivity check failed"
Root Cause Node has multiple NICs. The installer selected the wrong (inactive/unconfigured) one. This is common on VMware VMs that have a management NIC and a cluster NIC.
Fix Expand the host row in the UI → Find the NIC section → Deselect inactive NIC → Select working NIC → Click "Re-validate"
Outcome Node moved to Ready status within 60 seconds. This exact issue affected one of my nodes — it blocked me from starting installation until fixed.
SCENARIO C Node shows wrong hostname after discovery ▼
Symptom Node appears but hostname is auto-generated (e.g., localhost.localdomain) instead of your planned name
Fix Rename the host in the UI to match your DNS records (e.g., ocp-master1.shivang.lab.kuberox.net)
If ignored Post-install oc get nodes shows random hostnames not matching DNS. Authentication and ingress operators may have certificate issues as a result.
SCENARIO D SSH access fails after node boots from ISO ▼
Symptom Cannot SSH into nodes from bastion after booting from the discovery ISO
Root Cause Proxy certificate validation issue on the bastion side. The SSH key is correctly embedded in the ISO, but the bastion's proxy was interfering with the connection.
Workaround Access the node directly via VM console in vSphere while troubleshooting the proxy issue. Installation can proceed without SSH access to nodes.

Installation Phases — What Actually Happens

PHASE 01
Bootstrap
15:16 → ~16:00
A temporary bootstrap process starts an ephemeral API server to initialize etcd and the control plane. Progress bar moves slowly (~0–30%). This is normal.
PHASE 02
Control Plane
~16:00 → ~17:30
Masters download and install RHCOS to disk, reboot, and form etcd quorum. All three masters must join. Watch with oc get no,co.
PHASE 03
Worker Join
~17:30 → ~18:10
Workers install RHCOS and join the cluster. Workers failing here does NOT abort the cluster. Let it complete and add failed workers via Add Hosts.
PHASE 04
Operator Init
~18:10 → 19:05
All 30+ cluster operators deploy, download images, and stabilize. This is the longest and most confusing phase. Most operators self-heal within minutes.

Phase 3 Critical Decision: Worker Failure

At 17:39, my UI showed:

Assisted Installer UI — 17:39
Progress: 62%
Status: 1/3 workers failed — 1 pending user action
Failed host: ocp-worker3
🚨
CRITICAL DECISION POINT — Do NOT abort. A single worker failure does not stop the cluster. If you abort here you lose the entire installation and must wipe all nodes and start over. Let it complete. Add the worker afterward using the Add Hosts flow (see Section 07).

Every Issue I Hit — Complete Breakdown

ISSUE 1 NIC Connectivity Error on Host Discovery ▼
SymptomNode shows "Insufficient" or error state after booting from ISO
Root CauseWrong NIC selected — the NIC was inactive or misconfigured. Common on VMs with multiple network adapters.
FixExpand the host in the UI → deselect the inactive NIC → select the working NIC → click Re-validate
Outcome✅ Node moved to Ready within 60 seconds. Installation could proceed.
ISSUE 2 SSH Access Failed Post-Boot ▼
SymptomCould not SSH into the node from bastion after it booted from the ISO
Root CauseProxy certificate validation issue on the bastion side
FixWorkaround: access the node via VM console directly until the proxy issue is resolved. Installation can proceed without SSH.
Outcome✅ Used VM console as fallback. Did not block installation.
ISSUE 3 API Connection Timeout — oc Commands Failing ▼
Symptom"Unable to connect to server" on every oc command from the bastion
Root CauseThe Bastion VM was powered off. The kubeconfig points to the API VIP which routes through the network — but oc commands are issued from the bastion.
FixPower on the Bastion VM. Connectivity restored immediately. Keep bastion running throughout the entire installation.
Outcome✅ Commands worked immediately after powering on. Obvious in hindsight.
ISSUE 4 1 Worker Failed During Initial Install (ocp-worker3) ▼
SymptomUI showed "1/3 workers failed" at 62% progress during the installation process
Root CauseWorker node failed mid-installation — likely a disk timing issue or transient network blip during RHCOS write. Intermittent VMware storage latency spikes can cause this.
FixLet the main cluster complete — do NOT abort. Then use the "Add Hosts" tab to re-boot the failed worker with a new discovery ISO and re-install it separately. See Section 07 for the full procedure.
Outcome✅ Main cluster completed successfully at 19:05. Worker3 added at 20:27, joined at 20:50. Total delay: ~1h 45m.
ISSUE 5 Control Plane Nodes Briefly NotReady ▼
Symptomocp-master2 and ocp-master3 showed NotReady for a short period at 18:29
Root CauseMachine Config Operator triggered node reboots after workers joined the cluster to apply new configuration profiles. This is expected OCP behavior.
FixNone required. Wait 2–3 minutes. Self-resolving.
Outcome✅ Both masters returned to Ready automatically within 3 minutes.
ISSUE 6 Monitoring Operator Stuck PROGRESSING=True ▼
Symptomclusteroperator/monitoring showed PROGRESSING=True from 18:10 to ~18:50 — approximately 40 minutes
Root CauseThe Prometheus admission webhook deployment requires 2 worker replicas to reach quorum. With only 1 worker Ready, the webhook stayed Pending, blocking the entire monitoring operator.
FixNone required. Wait for the second worker to reach Ready state. Do not manually delete or restart monitoring pods — this will make it worse.
Outcome✅ Monitoring stabilized ~4 minutes after ocp-worker2 reached Ready state.

Final healthy state for all operators: AVAILABLE=True, PROGRESSING=False, DEGRADED=False. If any operator shows DEGRADED=True after 30 minutes post-installation, run oc describe co <operator-name> and look at the Conditions section.

Monitoring Operator Deep-Dive

The monitoring operator (Prometheus + Alertmanager + Grafana stack) is always last. Here's why:

The prometheus-operator-admission-webhook deployment requires 2 worker replicas to reach quorum. With only 1 worker Ready, the webhook stays in Pending state. This blocks the entire monitoring operator from becoming Available. Once your second worker hits Ready, monitoring stabilizes within ~4 minutes — no manual intervention needed.

Monitor operator health in real-time
# Watch everything — most useful command during install
watch -n 2 'oc get no,co'

# If monitoring stuck, check webhook
oc get pods -n openshift-monitoring | grep webhook

# Describe stuck operator for root cause
oc describe co monitoring

Adding a Failed Worker — Complete Procedure

The cluster completed successfully at 19:05 with 5 nodes. ocp-worker3 was listed under "Could not install 1 worker hosts." Here's the exact procedure I used to add it.

Step-by-Step Add Hosts Procedure

Step 1
Go to console.redhat.com → OpenShift → your cluster
Step 2
Click the "Add Hosts" tab at the top of the cluster detail page
Step 3
Click "Add hosts" — this generates a NEW discovery ISO (separate from the original)
Download this ISO. Do not reuse the original ISO from the initial installation.
Step 4
Mount the new ISO on the failed worker VM and boot from it
Step 5
Wait for the node to appear in the Host Discovery table with Ready status
Status transition: Discovering → Pending → Ready (typically 3–5 min)
Step 6
Click "Install ready hosts" — installation begins
Watch: Installing → Installed
Step 7
Verify the node joined the cluster
Verify + approve CSRs after Add Hosts
# Confirm node appears and is Ready
oc get nodes

# If node shows NotReady after appearing — check for pending CSRs
# Manually-added nodes don't get auto-approved CSRs
oc get csr

# Approve ALL pending CSRs at once
oc get csr -o name | xargs oc adm certificate approve

# Run the above command TWICE — there are usually two rounds of CSRs
# (node bootstrap CSR, then kubelet serving CSR)
⚡
CSR Pro Tip: For manually-added nodes, run the CSR approval command twice. There are typically two rounds — a bootstrap CSR first, then a kubelet serving certificate CSR after the node registers. Running it once may leave the node stuck in NotReady.

Everything I Learned — The Full List

01
DNS is everything
Most OCP failures trace back to DNS. Verify all records from the bastion — forward AND reverse — before clicking Install. Never from your laptop.
02
Never abort on worker failure
A single worker failure does NOT stop the cluster. Aborting does. Let it complete, use Add Hosts for the failed worker afterward.
03
Keep bastion running
The bastion must stay powered on throughout installation. Every oc command you issue comes from here. If it's off, nothing works.
04
NotReady is normal
Nodes reboot during MCO rollouts. NotReady for 2–3 minutes during or after worker join is completely expected. Only investigate at 10+ minutes.
05
Monitoring operator is always last
It needs 2 Ready workers for the Prometheus admission webhook. Don't touch it — just wait for your second worker.
06
Download kubeconfig immediately
The kubeadmin-password is shown once. Save the kubeconfig and kubeadmin-password to a safe location the moment installation completes.
07
Run CSR approval twice
When adding nodes manually, there are two rounds of CSRs. Run the approval command twice to catch both bootstrap and serving certificate requests.
08
Prepare IPs and MACs in advance
Build a spreadsheet with every node's static IP, MAC address, gateway, and DNS before starting. Hunting for MACs mid-setup introduces errors.
09
Use Minimal ISO
Always choose Minimal ISO. It's smaller, faster to download, and pulls remaining data from the network. Works perfectly with internet-connected nodes.
10
watch -n 2 'oc get no,co'
This single command is your complete real-time view of installation progress. Run it in a terminal and leave it open for the entire install.
